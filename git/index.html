Distributed

With Git you clone the whole history of the project from inception to current state

git clone username@host:/path/to/repository

After the cloning there is basically no information that you do not have compared to the server you cloned from

With SVN you checkout only a specific snapshot of the project

svn checkout https://path/to/repository

With Git you have the whole history locally

Repository from scratch

With Git you add files to the so called 'index' which is basically a staging area.

Git internals

Git is "content-addressable filesystem with a VCS user interface written on top of it"

- The key being the checksum of the content that you committed.

This checksum is actually a SHA1 hash digest of the content of the committed files with metadata(message, author, etc) and reference to the commit key before it.

- The content being an object containing the tree structure with a snapshot of the files that you committed.

key: 78c61dcff33b0cea7755eb71c8b188af37c97b48

content: index.html, css/style.css, js/site.js

Git stores the full content of the committed files(with references to unchanged files between versions).

It does not store just the deltas like SVN. This means some speed coming from the fact that when you pull out a version it's fast since it does not have to build up the files processing all the deltas.

Branching is 'cheap' with Git

40 bytes cheap, because a branch is just a pointer(bookmark) of the last commit in the branch(the SHA1 key)

Remember a git commit has a reference back to it's parent. This makes branching look like

With SVN you see branching as something "big" to be done when releasing a new version of the project or for a different brand/customer because others will see the branches you create and may not have meaning for them.

With Git since you have your own repository you can create whatever branches you want and you choose if to share it with others.

Git encourages you to create so called Topic branches

A "merge commit" has two parents so it keeps reference to the commits that led to it.

After the merge you will get also the commit messages from the commits that were in the two branches(ordered by time of the commits)

This allows to distinguish about who actually made the changes from which merged them.

You don't have to collaborate only with the repository from which you cloned.

At cloning time you get an easy reference to the repository you clone from by the name origin.

Moving away from giving "committer" status to a developer to a more pull changes that add value to a project.
